# Decisions

One line per decision, append-only, newest last.

Format: `YYYY-MM-DD: <decision>, because <reason>. Revisit if <condition>.`

---

- 2026-07-14: Build Parser Arena as a standalone project beside `syshin0116.dev`, because parser execution, uploads, generated artifacts, and deployment needs do not belong in the immutable blog repository. Revisit only if the blog later exposes a read-only integration surface.
- 2026-07-14: Create the document workspace immediately after upload and append parser runs one at a time, because users often need to inspect one result before deciding which parser to try next. Revisit if batch benchmark studies require a separate bulk-selection surface.
- 2026-07-14: Package each parser adapter as an independently pinned OCI image and keep the runner focused on orchestration, because parser dependencies conflict and evaluations must record reproducible execution environments. Revisit if a parser's supported hardware cannot run through OCI or its official immutable service interface is a stronger boundary.
- 2026-07-14: Make execution target a per-run runner choice behind one location-independent API, because private documents may stay on a local Docker host while MinerU and other GPU parsers can use a cloud Linux runner with the same pinned image contract. Revisit if the product becomes exclusively hosted or exclusively offline.
- 2026-07-14: Make the versioned parser adapter protocol and conformance suite the primary product boundary and limit the launch catalog to OpenDataLoader PDF, MinerU Pipeline, and LightOnOCR-2-1B, because these three exercise deterministic, structured pipeline, and VLM outputs without coupling the core to a large catalog. Revisit if a fourth parser cannot be added without core or UI changes.
- 2026-07-14: Supersede the parser-only adapter boundary with typed immutable artifacts and a fixed linear component recipe of preprocessors, one parser, postprocessors, and optional chunk/embed/vector sink, because parser outputs must later compose with interchangeable LLM and retrieval infrastructure without requiring a general DAG engine. Revisit if a demonstrated workflow requires branching or loops.
- 2026-07-14: Supersede the three-parser launch plan with an MVP that implements OpenDataLoader first and MinerU second while deferring LightOnOCR and the broad catalog, because one excellent vertical slice and one real comparison validate the product before more integrations. Revisit after the two-parser comparison and component extension test work end to end.
- 2026-07-14: Hide recipes, execution targets, versions, and provider configuration behind reviewed defaults and an Advanced disclosure, because first-time use should be upload, one-click run, evidence inspection, and optional comparison. Revisit if expert users cannot reproduce or customize runs through the advanced surface.
- 2026-07-14: Keep the MVP information architecture to one upload page and one stateful document workspace, with parser selection, run details, comparison, and Judge handled as workspace states or overlays, because users should not navigate away from the source while evaluating results. Revisit when accounts, batch studies, or reusable external connections require independent navigation.
- 2026-07-15: Wrap ingest, pipeline-run, and evaluation jobs in one thin shared LangGraph Functional API envelope while keeping component recipes linear and the domain database authoritative, because restartable orchestration should be available without turning the product into a workflow builder. Revisit if checkpoint recovery does not justify its duplicate operational state or a concrete workflow needs explicit graph topology.
- 2026-07-15: Use PostgreSQL for authoritative domain jobs and a separately owned LangGraph checkpoint schema in reference Docker and hosted deployments, because combining D1 domain state with PostgreSQL checkpoints would add an unnecessary metadata system. Revisit if LangGraph checkpointing is removed or a serverless-only deployment proves D1-only operation is required.
- 2026-07-15: Depend on a small S3-compatible BlobStore contract, with SeaweedFS as the reference open-source self-host store, Cloudflare R2 for the hosted service, and a local directory for minimal development, because the product should not depend on AWS S3 or one storage implementation. Revisit if either production provider fails the byte-range, signed URL, deletion, or restart conformance suite.
- 2026-07-15: Use PDF.js for interactive source viewing, create pinned PDFium page images only as explicit auxiliary artifacts, and expose only parser-native geometry in the MVP, because source evidence must remain truthful and deterministic without making rasterization mandatory. Revisit when derived alignment has its own visible provenance and evaluation contract.
- 2026-07-15: Integrate OpenDataLoader PDF 2.5.0 in deterministic native-digital-PDF mode as the first runnable OCI parser, preserving raw JSON and Markdown while normalizing only parser-native bounding boxes, because this validates the component boundary without coupling parser code to the web app or workflow runtime. Revisit if upstream exposes truthful partial artifacts or the upload workspace requires a service-integrated execution path.
- 2026-07-15: Standardize JavaScript and TypeScript dependency management, scripts, locks, and tests on Bun, and require each future Python service or extension to own a uv-managed `pyproject.toml` and `uv.lock`, because independently deployable components need fast reproducible toolchains without npm or cross-component Python dependency coupling. Revisit if a required upstream package cannot pass the component smoke suite under these installers.
- 2026-07-15: Make `make dev` the host HMR path and `make up` the detached health-checked Compose path, with the default Compose stack containing only runnable services and no web-mounted Docker socket, because onboarding should be one command without pretending planned infrastructure is implemented or weakening parser isolation. Revisit when the upload control plane can call a real runner service and the first persistent service boundary lands.
- 2026-07-15: Vendor the five `emilkowalski/skills` design-engineering skills at commit `7bb7061` under `.agents/skills`, because UI design and motion guidance should be project-local, reviewable, and reproducible instead of depending on a mutable global installation. Revisit when intentionally upgrading upstream or if project-local skill discovery changes.
- 2026-07-15: Pin the client-only interactive renderer to `react-pdf` 10.4.1 over `pdfjs-dist` 5.4.296 with its matching locally bundled worker, while keeping parser-native evidence in a separate overlay and server PDFium renders as explicit auxiliary artifacts, because the MVP needs controlled page and text rendering without coupling viewer lifecycle to parser geometry or evaluation images. Revisit if the wrapper blocks required range, virtualization, or overlay coordination, or a security or correctness fix requires upgrading the pinned pair.
- 2026-07-15: Vendor the pinned PDF.js CMaps, standard fonts, WASM codecs, and ICC profile under a versioned public path, because Korean, legacy-font, and JPEG 2000 documents must render reproducibly without a mutable third-party CDN. Revisit when the build can copy and verify these exact-version assets without adding a provider-specific runtime path.
- 2026-07-16: Compose each run candidate from per-stage slot selections, with a required parser/OCR slot and optional later postprocessor slots defaulting to none, and key comparison, votes, caching, and lineage on the full slot-combination fingerprint, because candidates will grow into parser-plus-LLM combinations and unchanged stage results must be reusable across combinations without rerunning earlier stages. Revisit if a demonstrated workflow requires non-linear composition.
- 2026-07-16: Surface each parser in the slot dropdown as a small set of curated named profiles and render the underlying option set from the manifest-declared option schema under Advanced, with resolved normalized options always part of the fingerprint and reproduction manifest, because parsers expose large option surfaces that must stay reviewable and reproducible without parser-specific UI code. Revisit if a parser's options cannot be expressed as a declarative schema.
- 2026-07-16: Offer blind comparison as a workspace display mode that masks candidate labels and shuffles column order until the user submits a vote using the same anonymized listwise schema as the LLM Judge, and record on every human vote whether it was blind, because unbiased preference capture and label-visible inspection should share one workspace and only blind votes may feed any future community ranking. Revisit when a public community arena surface is actually scheduled.
- 2026-07-16: Split identity into a candidate-spec fingerprint, per-stage invocation fingerprints, and artifact content hashes, refining the earlier single-fingerprint wording, and publish cache entries only after validation under a canonical-encoding spec with test vectors, because one hash cannot serve candidate identity, stage reuse, and vote history simultaneously and inconsistent option serialization would silently break caching. Revisit if the three-layer model proves heavier than the linear recipe justifies.
- 2026-07-16: Never automatically reuse outputs of stages whose manifest declares nondeterminism, offering an explicit reuse-versus-rerun choice and labeling reused samples as cached observations, because presenting one LLM sample as a fresh execution misrepresents variance. Revisit if seeded determinism becomes universally available for hosted models.
- 2026-07-16: Store on every human vote the exact candidate artifact ids, the displayed permutation, and the session's blind-exposure state, because votes must stay historically meaningful after reruns and cache reuse and position bias must be self-auditable. Revisit only alongside a versioned vote-schema change.
- 2026-07-16: Serve in three phases with the web control plane on the edge runtime throughout: a device-local demo with no server parsing first, then one reference VM running the CPU parser through the same Compose stack as self-hosting, then serverless per-second GPU runners plus bring-your-own-runner connections, because the three workload shapes have incompatible hosting economics and the reference deployment should double as operations documentation. Revisit at each phase boundary or if the experimental Vinext runtime blocks the hosted path.
- 2026-07-17: Serve the slot dropdown from versioned catalog entries that add registry review metadata (pinned image digest, maturity, per-deployment availability, named profile revisions, license review) over each component manifest, with OCR and VLM parsers expressed as parser capabilities rather than separate entry types and concrete LLM choice living in the user's connection rather than the catalog, because the UI must render what was reviewed without reading extension directories and the catalog must not multiply entry kinds per model family. Revisit if a component class cannot be described by role plus capabilities.
- 2026-07-17: Organize the user-facing product as three surfaces — the document workspace, a blind Arena battle with sample-document cold start, and a per-document-type Leaderboard fed only by blind votes — and defer the LLM Judge until human blind voting exists, because human votes need no provider keys or consent boundary, they seed the leaderboard, and a later Judge can then be validated against human agreement. This supersedes both the workspace-only information architecture and the Judge-in-MVP scope. Revisit if arena participation stays too low to rank anything.
- 2026-07-19: Pin the MinerU extension to 3.4.4 under the MinerU Open Source License instead of the AGPL-3.0 2.7.6 line, because upstream removed the AGPL DocLayout-YOLO code dependency in 3.x and the Apache-based terms (attribution plus 100M-MAU/USD-20M thresholds) are compatible with a hosted catalog entry, while AGPL obligations were not reviewable in time. Revisit if the pipeline output contract or model-weight licenses change on upgrade.
- 2026-07-19: Add Azure Document Intelligence as the first remote parser, executed as an OCI component that declares network: remote plus a connection whose endpoint/key the runner injects from a local .env (never into artifacts or logs), because a cloud OCR that is strong on Korean is worth the first outbound-network exception while the connection-and-network-policy pattern gets exercised end to end. Revisit when a hosted deployment needs signed per-run credentials instead of a static local key.
- 2026-07-19: Keep parser-native geometry the default evidence and offer merging only as an explicit per-view toggle (Native = each parser box as reported, Merged = a pure geometric union of one table's cells or one Azure DI line's words), because the product's job is to show what the parser actually reported, and joining is a convenience that must never be mistaken for the raw output or for text-to-coordinate inference. Revisit if a parser needs a merge grouping beyond table and line scope.
- 2026-07-19: Supersede the edge-runtime-throughout portion of the 2026-07-16 deployment plan by using the official Next.js Node.js runtime for the web and short-lived control-plane routes, with Vercel as the first hosted target and durable orchestration plus OCI parsing deployed separately, because Vinext remains experimental and the growing authentication, PostgreSQL, signed-upload, and job APIs benefit from the verified Next.js toolchain. Revisit if a verified edge adapter reaches full compatibility or measured latency requires the control plane at the edge.
- 2026-07-20: Show each pinned upstream option surface in full from JSON Schema and annotate fixed or currently unavailable properties and choices under `x-parser-arena` with a reason and source URL, because experts need to inspect real capabilities without allowing the current runner to submit unsupported values or requiring parser-specific UI code. Revisit if standard JSON Schema annotations can express environment-specific availability and per-choice reasons without this extension.
- 2026-07-20: Rename the product and repository from Parser Arena to Document Arena and atomically move the unpublished v1alpha1 contract namespace, schema annotations, media types, request headers, package names, and image labels to `document-arena`, because the fixed linear component model is intended to evaluate preprocessing, OCR/parsing, postprocessing, chunking, embedding, indexing, retrieval, and eventual RAG pipelines rather than parsers alone. Preserve the original IndexedDB and localStorage keys so existing device-local documents, results, and votes remain visible, and accept legacy runner environment variable names during the transition. Revisit only before a stable contract release or if an adopted external extension requires an explicit compatibility layer.
- 2026-07-20: Accept user-supplied remote-provider credentials through a generic manifest-driven Connections page but keep UI-entered secrets only in the loopback runner's memory, with ignored `.env` values as a read-only developer fallback, because the local prototype has no trustworthy encrypted secret store and credentials must never enter browser persistence, URLs, options, artifacts, logs, or Docker process arguments. Revisit when a platform keychain or hosted secret manager can provide named persistent connections with auditable access controls.
- 2026-07-20: Stop vendoring agent-specific design skills and ignore `.agents/` while retaining `AGENTS.md` and this decision log as repository-level contributor guidance, because local assistant tooling is not a product or runtime dependency but the product contract and hard-to-reverse choices must remain public and reviewable. This supersedes the 2026-07-15 skill-vendoring decision. Revisit if a project-local agent skill becomes a required, versioned contributor tool.
- 2026-07-20: Normalize Azure Document Intelligence output directly from the SDK's native line, word, paragraph, table, span, and polygon models instead of aligning rendered Markdown text back to words, because span-linked parser geometry is more truthful, deterministic, and independently maintainable. Revisit if the pinned service contract stops returning stable native spans or polygons for a required content type.
