# Durable workflow orchestration

Status: accepted architecture boundary, local-first retention updated 2026-07-20
Date: 2026-07-15

## Decision

Document Arena wraps the three long-running product lifecycles—document ingest,
each pipeline run, and each evaluation run—in one shared thin LangGraph workflow
envelope. LangGraph checkpoints the execution cursor and
allows a job to resume after an orchestrator restart. It does not define the
product domain, artifact schema, parser catalog, queue lease, cancellation,
public event log, or user-visible pipeline topology.

There are three job-specific entrypoints over the same envelope and task
library:

~~~text
ingest:{ingestJobId}
pipeline:{pipelineJobId}
evaluation:{evaluationJobId}
~~~

A document workspace is not one permanent graph thread. It is the domain object
that aggregates completed and active jobs. Every opaque job id belongs to one
owner and has at most one active lease. This keeps parser runs independent,
preserves re-ingest history, and avoids contention when a user starts more than
one parser.

## Responsibility boundary

| Concern | Owner |
|---|---|
| Retained documents, workspaces, runs, and artifact index | Browser IndexedDB |
| Retained source and large artifact bytes | Browser OPFS (IndexedDB blob fallback) |
| Jobs, attempts, leases, events, and transfer deadlines | Domain database |
| Queue claim, lease, heartbeat, cancellation request | Domain job worker |
| Temporary remote-execution PDF and generated bytes | BlobStore |
| Rebuildable resume cursor | LangGraph checkpointer |
| Component execution, process kill, and resource limits | Generic runner |
| Parser-specific invocation and normalization | Component extension |
| User-facing workspace state | Browser application derived from IndexedDB/OPFS |

IndexedDB/OPFS is the system of record for the user's retained workspace and
results. The domain database is the system of record for server-side execution
jobs. LangGraph checkpoints are operational state and are never the only copy
of a user-visible result or authoritative job transition.

The API and job worker never read checkpoints as product state. The UI and
exports read retained content from IndexedDB/OPFS. LangGraph Store is not used
for product memory or artifact metadata; normal domain tables provide only
cross-workflow operational data.

## System flow

~~~mermaid
flowchart LR
    B[(Browser IndexedDB + OPFS)] --> API[Web gateway]
    API --> D[(Domain database)]
    API --> O[Workflow service]
    O --> D
    O --> C[(LangGraph checkpoints)]
    API -. presigned URLs .-> B
    B <--> A[Temporary BlobStore]
    O --> R[GCP generic runner]
    A <--> R
    R --> X[Component OCI container]
    X -. output bundle .-> R
    R -. receipt .-> O
    O --> E[Evaluator adapter]
    E -. result .-> O
~~~

The web gateway creates a queued `Job` and outbox row in one domain transaction.
A dispatcher claims it with a time-bounded lease and starts the deterministic
thread id. Delivery is at-least-once: a crashed or stale worker may dispatch the
same job again, so every task must short-circuit from authoritative domain
records or reuse the same external idempotency key.

The domain transaction and checkpoint write are intentionally not treated as
one distributed transaction. A checkpoint may lag and be rebuilt; a domain
record may not.

## Ingest workflow

For hosted execution, the browser stages a local source through a short-lived
presigned PUT, then calls the source-complete endpoint. The server verifies
object HEAD, size, media type, and checksum before its domain transaction queues
the execution outbox record. The R2 object remains temporary; the original in
IndexedDB/OPFS stays authoritative.

~~~text
verify_source
  -> hash_and_inspect
  -> persist_temporary_source_manifest
  -> optional_first_thumbnail
  -> mark_execution_input_ready
~~~

The upload body and presigned URL are never stored in graph state. The state
contains the opaque local workspace id and provider-neutral temporary source
reference only.

## Pipeline run workflow

Each parser selection creates a new PipelineRun and a new LangGraph thread.

~~~text
resolve_recipe
  -> prepare_next_stage
  -> submit_stage
  -> await_stage
  -> validate_bundle
  -> publish_temporary_stage_artifacts
  -> more stages? ---- yes ---> prepare_next_stage
  -> no
  -> mark_result_ready_for_browser_import
~~~

The graph is generic. It executes the component manifest resolved for the
current stage and never routes by parser id. The stable stage key is derived
from `runId`, stage index, and resolved recipe hash.

The recipe remains a typed linear sequence:

~~~text
preprocess* -> exactly one parser -> postprocess* -> chunk? -> embed? -> sink?
~~~

LangGraph wraps the lifecycle of that sequence. It does not turn component
recipes into arbitrary graphs, and the MVP UI does not expose branches, loops,
or a node editor.

Long container work is split into submit and await boundaries. The runner
returns a stable runnerJobId; resuming submit with the same run and stage
idempotency key returns the existing job rather than starting another container.

Publishing an output is also idempotent:

1. upload to a job-scoped immutable temporary key;
2. calculate and verify SHA-256, size, media type, and artifact schema;
3. register the transfer record and successful attempt in one domain
   transaction;
4. reuse that successful `StageRun` when the workflow replays;
5. remove unregistered temporary objects with an orphan sweeper.

The browser then downloads through a newly authorized presigned GET, verifies
the receipt and hashes, and atomically imports the result into IndexedDB/OPFS.
Import acknowledgement schedules explicit job-prefix deletion. The unfiltered
one-day R2 lifecycle rule is a backstop for missing acknowledgements and failed
cleanup, not durable retention.

An S3 ETag is transport metadata and must not be treated as a SHA-256 checksum.
An external LLM or vector sink receives its own provider request/idempotency key.

## Evaluation workflow

Evaluation is a separate read-only workflow because it may consume several
candidate artifacts.

~~~text
resolve_candidates
  -> deterministic_metrics
  -> optional_evidence_render
  -> optional_llm_judge
  -> aggregate
  -> publish_evaluation
~~~

The first Judge is one fixed pass. Later, only this workflow may add parallel
rubrics, disagreement checks, human approval, or iterative evidence selection.
It never mutates a parser artifact.

## Minimal workflow state

Graph state contains identifiers and small control values:

~~~json
{
  "workflowVersion": "v1",
  "jobId": "job_...",
  "documentId": "doc_...",
  "runId": "run_...",
  "recipeRevision": "sha256:...",
  "currentStageIndex": 0,
  "inputArtifactIds": ["artifact_..."],
  "outputArtifactIds": [],
  "runnerJobId": null,
  "cancelRequested": false
}
~~~

Do not checkpoint:

- PDF or page-image bytes;
- full parser JSON or Markdown;
- document text sent to a Judge;
- large geometry arrays;
- secrets, signed URLs, or local paths.

Retained values live in browser IndexedDB/OPFS; in-flight values may live in the
temporary `BlobStore`; secrets remain in the runtime secret manager. Presigned
URLs are generated when needed and never persisted. Long-lived BlobStore
credentials never enter browser or GCP job state.

## Persistence

- Local workflow experiments may use SqliteSaver.
- Docker and hosted production use PostgresSaver.
- Domain and checkpoint tables may share one PostgreSQL cluster but use separate
  schemas, migrations, credentials, and retention rules.
- Every workflow invocation supplies its job-scoped deterministic thread id.
- PostgresSaver is not the queue, worker lease, heartbeat, cancellation channel,
  or public event stream.
- Deleting a local workspace removes its IndexedDB/OPFS state and requests
  cancellation for any active execution. An outbox-driven idempotent cleanup
  job deletes its temporary object prefix and related checkpoints with a
  retryable cleanup state; the R2 one-day lifecycle remains the orphan
  backstop.

Checkpoint replay must not create duplicate side effects. Every external task
uses an idempotency key and writes immutable artifact keys.

Checkpoint state, errors, and metadata follow the same no-document-content rule
as graph state; exception serialization must not leak extracted text or signed
URLs.

## Status, progress, and cancellation

Nodes write authoritative stage and run transitions plus an append-only
`JobEvent` in the same domain transaction. Events have a monotonic per-job
`seq`. The UI polls with `afterSeq`; a later SSE endpoint can replay the same
records with `Last-Event-ID`. Custom LangGraph stream events and container
stdout are diagnostic inputs, not the public event API.

The complete internal job states are:

~~~text
queued -> running -> completed | failed
             |
             -> cancel_requested -> cancelling -> cancelled
~~~

Cancellation is idempotent and cooperative:

1. atomically mark cancellation requested in the domain database;
2. notify the active workflow and stop scheduling new stages;
3. make the runner kill the active OCI process;
4. clean the attempt work directory and any unpublished object;
5. use a compare-and-set so completion and cancellation races produce one
   terminal state;
6. preserve any already imported local artifacts, label the run cancelled, and
   delete unpublished temporary objects.

One failed or cancelled parser workflow does not alter another parser workflow.

## Implementation shape

Start with one LangGraph Functional API envelope and idempotent tasks. Each job
kind supplies its small ordered task function. Move a job kind to explicit
StateGraph nodes only when conditional edges, parallel workers, or human
interrupts are real requirements.

~~~text
services/orchestrator/
  workflows/
    job_envelope
    ingest
    pipeline_run
    evaluation
  jobs/
    dispatcher
    leases
    events
  tasks/
    artifacts
    runner
    evaluation
  persistence/
~~~

LangGraph remains behind a small WorkflowGateway used by the control plane. The
component protocol, artifact schemas, and runner API must remain usable without
importing LangGraph.

## Retry semantics

- Retrying a failed technical attempt with the same resolved recipe stays under
  the same `PipelineRun` and creates a new `StageAttempt`.
- Replaying a completed stage returns its registered immutable artifacts.
- Changing options, component/image/model revision, or prompt creates a new
  `PipelineRun` and workflow job.
- API creation accepts an owner-scoped `Idempotency-Key` backed by a unique
  constraint, so a client retry does not create a duplicate document, run, or
  evaluation.

## Conformance checks

- Restart the orchestrator during every workflow phase and verify resume.
- Crash after the queued-job/outbox transaction but before dispatch and verify
  replay starts one job-scoped workflow.
- Expire a worker lease and verify one replacement worker safely resumes it.
- Invoke submit_stage twice and verify only one runner job exists.
- Replay a completed checkpoint and verify immutable artifacts are reused.
- Crash after domain commit but before checkpoint and verify the task
  short-circuits without duplicate side effects.
- Cancel while waiting for a parser, verify the OCI process stops, and verify no
  later stage starts.
- Race cancel against completion and verify exactly one terminal state.
- Poll events from a saved sequence and verify ordered replay without gaps.
- Fail one parser while another succeeds on the same document.
- Delete a document and verify artifacts plus checkpoint metadata are removed.
- Add a fixture component without changing graph topology.
